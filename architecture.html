Video Streaming — HLS & MPEG-DASH
Adaptive bitrate streaming with encrypted segments

All video content is transcoded into HLS (.m3u8 + .ts segments) and optionally MPEG-DASH format using an FFmpeg pipeline. Videos are never served as direct MP4 files — they are split into short encrypted segments that are meaningless without the decryption key.

📦
Segment Duration
2–6 second segments. Shorter segments = finer key rotation granularity.
📶
Adaptive Bitrate
Multiple quality tracks: 360p, 480p, 720p, 1080p, 4K. Client auto-selects based on bandwidth.
🔐
AES-128 per segment
Each segment encrypted. Keys rotate every N segments. Keys only served after auth.
⚡
CDN Delivery
Segments cached on CloudFront edge nodes globally. Fast + signed URLs.
FFmpeg — Transcode to multi-bitrate HLS with AES-128 # Transcode single source to ABR HLS with AES-128 encryption ffmpeg -i input.mp4 \ -filter_complex \ "[0:v]split=4[v1][v2][v3][v4]; \ [v1]scale=1920:1080[h1080]; \ [v2]scale=1280:720[h720]; \ [v3]scale=854:480[h480]; \ [v4]scale=640:360[h360]" \ -map "[h1080]" -map "[h720]" -map "[h480]" -map "[h360]" -map 0:a \ -c:v libx264 -crf 22 -preset slow -profile:v high \ -c:a aac -b:a 128k \ -hls_time 4 \ -hls_key_info_file /keys/aes.keyinfo \ -hls_playlist_type vod \ -hls_segment_filename "seg_%v_%03d.ts" \ -master_pl_name master.m3u8 \ output_%v.m3u8
HLS Key Info File (aes.keyinfo) # Line 1: URL where player fetches the key (your secure key server) https://api.ksmovies.com/stream/aes-key/MOVIE_ID/KEY_ID # Line 2: Local path to the binary AES-128 key (16 bytes) /keys/aes128.key # Line 3: IV (initialization vector) — hex, rotated per batch 0x0000000000000000000000000000001

DRM — Multi-DRM System
Widevine · FairPlay · PlayReady — covers all browsers and devices

DRM (Digital Rights Management) is the gold standard for content protection. It ensures that decryption keys never exist in the clear — they are secured inside a hardware or software Trusted Execution Environment on the user's device.

DRM System Platform Security Level Robustness
Google Widevine Chrome, Firefox, Android, Smart TVs L1 (hardware) / L3 (software) L1: Highest — TEE enforced
Apple FairPlay Safari, iOS, macOS, Apple TV Hardware + Secure Enclave Very high — Apple ecosystem
Microsoft PlayReady Edge, Windows, Xbox SL3000 (hardware) / SL2000 SL3000: Hardware-enforced
Shaka Player — DRM Configuration (Frontend) shakaPlayer.configure({ drm: { servers: { 'com.widevine.alpha': 'https://api.ksmovies.com/drm/widevine', 'com.microsoft.playready': 'https://api.ksmovies.com/drm/playready', }, advanced: { 'com.widevine.alpha': { videoRobustness: 'SW_SECURE_CRYPTO', // L3 minimum audioRobustness: 'SW_SECURE_CRYPTO', } } } });
💡
License Server Setup For production, use EZDRM, BuyDRM, or Axinom as your multi-DRM license provider. These services handle Widevine, FairPlay, and PlayReady licenses from a single integration. Your backend acts as a proxy that validates the user's subscription before forwarding the license request.

AES-128 Encryption — HLS Segments
Every video segment encrypted — keys served securely

For platforms that don't support full DRM (e.g. some Android WebViews), AES-128 HLS encryption provides a strong second layer. Each segment is encrypted, and the decryption key is only available from our key server after authentication and subscription validation.

  • 1
    Key Generation A unique 128-bit AES key is derived per movie using HMAC-SHA256 from a master secret. Keys rotate every N segments (configurable).
  • 2
    Key Storage Keys are never stored in plaintext. They are derived on-demand using: HMAC(masterSecret, movieId:keyId)
  • 3
    Key Request by Player HLS player reads #EXT-X-KEY URI from playlist and requests the key. Your server validates JWT token before responding.
  • 4
    Key Response Server returns raw 16-byte binary key. Response headers include Cache-Control: no-store to prevent caching.
  • 5
    Key Rotation Every 50 segments (≈200 seconds), a new keyId is used. Even if one key is leaked, attacker only gets ~3 minutes of footage.

JWT Token Authentication
Signed session tokens with expiry and plan embedding

Every video stream request must include a valid JWT session token. The token is issued by the backend after verifying the user's Firebase session and subscription status. Tokens expire after 4 hours, matching a typical movie viewing session.

JWT Payload Structure { "uid": "firebase-user-uid", "email": "[email protected]", "plan": "premium", // basic | standard | premium "iat": 1709481600, // issued at "exp": 1709496000, // expires in 4 hours "iss": "ksmovies.com", "aud": "ks-player" }
Token Validation Flow (Backend Middleware) async function verifyToken(req, res, next) { const token = req.headers['x-ks-token']; if (!token) return res.status(401).json({ error: 'No token' }); // 1. Check revocation cache if (tokenCache.get(token) === 'invalid') return res.status(401).end(); // 2. Verify JWT signature + expiry const payload = jwt.verify(token, JWT_SECRET); // 3. Confirm Firebase account still active const user = await admin.auth().getUser(payload.uid); if (user.disabled) return res.status(401).end(); req.user = payload; next(); }

Signed URLs & CDN Security
CloudFront signed URLs prevent hotlinking and sharing

Video segments are not served from open URLs. Each streaming session generates a signed URL that encodes the expiry time, allowed IP range, and a cryptographic signature. An attacker who copies the URL cannot use it from a different IP or after it expires.

ParameterValuePurpose
Expiry4 hours from generationURL becomes invalid after session ends
IP RestrictionUser's IP at session startURL unusable from different network
SignatureRSA-SHA1 with private keyTamper-proof — cannot forge URL
Session IDUnique per playback sessionEnables per-session revocation
Cache-Controlprivate, max-age=300Prevents CDN from caching segments publicly
⚠️
IP Restriction Caveat Mobile users may switch IPs during playback (4G ↔ WiFi). Consider allowing the full /24 subnet instead of a strict /32 to balance security with usability.

Dynamic Watermarking System
Invisible user-ID watermark that moves position every 25 seconds

KS Movies uses a two-layer watermarking system — a client-side visual watermark for deterrence, and a server-side forensic watermark embedded into video frames for legal identification of piracy sources.

👁️
Visual Watermark
User ID + email fragment overlaid on video with 8–15% opacity. Position changes every 25 seconds. Captures recording attempts.
🔬
Forensic Watermark
UserID embedded in video pixel data using A/B variant streams (e.g. Irdeto Cloakware). Survives compression and screen recording.
Client-side watermark positioning (Player JS) function moveWatermark() { const x = Math.floor(Math.random() * (window.innerWidth - 200)); const y = Math.floor(Math.random() * (window.innerHeight - 60)) + 40; watermark.style.left = x + 'px'; watermark.style.top = y + 'px'; watermark.style.color = `rgba(255,255,255,${0.08 + Math.random() * 0.07})`; } // Move position every 25 seconds setInterval(moveWatermark, 25000);
✅
Forensic Watermark Recommendation For maximum protection, use Irdeto Cloakware or NAGRA NexGuard — these embed an imperceptible per-user watermark directly into video bitstream at the CDN level. Even if pirated, the source user can be identified.

Screen Recording Protection
Best-effort detection and playback pausing
🛡️
Browser Limitation Web browsers do not expose screen recording APIs to JavaScript. True screen recording prevention requires native apps with OS-level APIs (EME, HDCP). On web, best-effort measures are used — the watermark is the primary forensic tool.
🖥️
Tab Visibility Pause
Playback pauses when tab loses focus. Reduces usefulness of recording on hidden tabs.
🖨️
Print Blackout
CSS @media print hides all content. Print screen / OS screenshot shows only a warning message.
🔒
EME / HDCP (Native)
Widevine L1 + HDCP 2.2 prevents capture at hardware level on Chromecast, Android TV, Smart TVs.
🐞
DevTools Block
F12 / Ctrl+Shift+I blocked. DevTools window size detection pauses video and logs the event.

Deployment & Infrastructure
Production-ready cloud stack
☁️
AWS CloudFront CDN
Signed URLs + signed cookies. Edge caching of encrypted segments. Origin Shield protects S3.
📦
AWS S3 (Origin)
Private bucket. No public access. Only CloudFront OAC can read. Server-side encryption at rest.
⚙️
API — Node.js / PM2
Express server behind Nginx reverse proxy. PM2 cluster mode. Auto-restart on crash.
🔥
Firebase
Auth, Realtime DB for sessions/logs. Cloud Functions for alerts. Firestore for movie metadata.
🎬
FFmpeg Pipeline
AWS MediaConvert or self-hosted FFmpeg. Outputs multi-bitrate HLS to S3 on upload.
🔑
AWS KMS
Master encryption keys stored in KMS. AES keys derived on-demand. No key ever in plaintext storage.

Admin Security Controls
Real-time access management and piracy response
  • →
    Instant Access Revocation POST /admin/revoke-user disables the Firebase account AND kills all active streams in < 1 second. No token cache refresh needed.
  • →
    IP Blocking POST /admin/block-ip adds IP to in-memory blocklist (instant) and Firebase (persistent). Duration configurable (default 24h).
  • →
    Live Stream Monitor Admin panel shows real-time active sessions from Firebase Realtime DB listeners. See who is watching what, from where.
  • →
    Security Event Dashboard All security_events from Firebase displayed in admin panel. Filter by type, user, IP, or time range.
  • →
    Content Access Control Per-movie accessLevel field controls which plan tier can watch. Change in Firebase → takes effect on next stream request.
